# RDW v0.3 Workflow — Design

- **Date:** 2026-07-07
- **Status:** Implemented
- **Builds on:** v0.2 hardening (config router, strict validation, safe writes)

## Goals

1. Turn planned runs into a lightweight workflow system without becoming an LLM runner.
2. Export public JSON Schemas for editor/CI validation.
3. Add optional, provider-neutral adapter hooks for external runtimes.

## Non-goals

- No built-in model API calls in the core CLI.
- No autonomous research/drafting execution.
- Provider extras remain placeholders; a deterministic fixture adapter now
  proves the integration surface without external calls.

## #8 Lifecycle

### Task statuses

| Status | Meaning |
| --- | --- |
| `planned` | Prompt bundle emitted; no pipeline work recorded |
| `research-done` | Knowledge packet saved |
| `draft-done` | Domain draft written |
| `qa-passed` | QA cleared blockers |
| `qa-failed` | QA blocked; optional `--reason` |
| `final-done` | Humanizer/final copy complete |

### Commands

```bash
rdw status .rdw-runs/lis-leaderboard
rdw task mark research-done .rdw-runs/lis-leaderboard
rdw task mark qa-failed .rdw-runs/lis-leaderboard --reason "unsupported claim"
rdw batch status .rdw-runs/demo-batch
rdw batch resume .rdw-runs/demo-batch
```

### Artifacts

- `status.json` gains `updated_at`, `history[]`, and optional `reason`.
- Batch runs sync `summary.yaml` counts and append `batch-log.jsonl` on mark.

## #9 Schema export

```bash
rdw schema packet --format jsonschema
rdw schema batch --format jsonschema
rdw schema task-contract --format jsonschema
rdw schema packet -o packet.schema.json
```

Schemas are derived from the imperative validator and planner contracts. They are a public contract for editors and third-party tools, not a runtime requirement.

## #10 Adapters

```bash
pip install research-domain-writing[local]
pip install research-domain-writing[openai]
pip install research-domain-writing[anthropic]
pip install research-domain-writing[adapters]

rdw adapter list
rdw adapter run local .rdw-runs/lis-leaderboard
rdw adapter run openai .rdw-runs/lis-leaderboard --dry-run
```

- `TaskAdapter.run(run_dir)` reads `task-contract.yaml` + `status.json`.
- Stubs write `adapter-<name>.json` describing the handoff.
- `rdw task execute --fixture <fixture.yaml> <run-dir>` stages a deterministic
  receipt and artifacts, validates the packet and QA gate, and advances one
  task through the existing lifecycle. The fixture is a prototype, not a
  provider integration.
- The deterministic core remains provider-neutral; adapters are optional.

## Boundary preserved

RDW still does not browse, call model APIs, or draft autonomously. Lifecycle commands record state; adapters document runtime handoff; agents execute prompts.
