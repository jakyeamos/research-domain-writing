# ADR-001: Provider-neutral single-task adapter contract

- Status: Accepted for the next implementation slice
- Date: 2026-07-15
- Scope: one planned task run and its external research/writing runtime handoff

## Decision

RDW will use an artifact-first, provider-neutral adapter protocol for executing
one planned task at a time.

The deterministic RDW core owns planning, contract validation, canonical
artifacts, lifecycle transitions, batch projections, review gates, and
promotion into final output locations. An adapter owns the external runtime:
it may invoke a local agent, a subprocess, or a future provider integration,
but it must return a durable receipt and stage all produced artifacts inside a
namespaced attempt directory. The core validates that receipt and the staged
artifacts before changing canonical lifecycle state or promoting any output.

This decision extends the existing `TaskAdapter.run(run_dir)` seam. It does
not add a provider SDK, browsing, model calls, hosted persistence, or an
autonomous batch runner to the core.

## Context

The current adapter surface is deliberately a handoff stub:

- `src/rdw/adapters/base.py` accepts a planned `run_dir` and returns an
  `AdapterResult`.
- `src/rdw/adapters/stubs.py` validates the task contract and writes a small
  `adapter-<name>.json` status artifact without calling a model API.
- `src/rdw/planner.py` creates `task-contract.yaml`, `prompt-bundle.md`, and
  `status.json` as deterministic inputs for an external agent.
- `src/rdw/lifecycle.py` owns legal status transitions, `summary.yaml`, and
  append-only batch events.
- `src/rdw/io.py` provides atomic text writes and fsynced JSONL appends.

The next adapter implementation needs a stable trust boundary. Without one,
an external runtime could silently mutate canonical state, publish incomplete
work, leak credentials into artifacts, or make retries duplicate side effects.
The boundary must remain useful for the current local recorder and for a future
provider integration without making either one the architecture.

## Contract

### Core-to-adapter request

The core will provide a typed request containing:

- `run_dir`: the existing planned task directory.
- `task_contract`: the validated contents of `task-contract.yaml`.
- `prompt_bundle_path`: the read-only prompt bundle generated by the planner.
- `status_path`: the read-only current task status path.
- `adapter`: the selected adapter name.
- `attempt_id`: a unique identifier for this execution attempt.
- `idempotency_key`: a stable key for the same task inputs, adapter, and
  requested stages.
- `requested_stages`: the stages the external runtime is asked to perform.
- `dry_run`: whether external side effects are prohibited.

The request is a handoff, not permission to mutate the run directory. Secrets
and credential values must never be serialized into the task contract, prompt
bundle, request, receipt, or lifecycle logs.

### Adapter-owned attempt directory

Each invocation receives this staging root:

```text
<run_dir>/adapter-runs/<adapter>/<attempt_id>/
```

The adapter may create or replace files only inside its own attempt directory.
Artifact paths in the receipt are relative to that directory. The executor
must reject absolute paths, `..` traversal, symlink escapes, and artifacts
outside the declared attempt root. An adapter must not write
`status.json`, `summary.yaml`, `batch-log.jsonl`, knowledge directories, or
final output directories.

The current `adapter-<name>.json` files remain compatibility summaries during
migration. They are not the authoritative execution receipt.

### Adapter receipt

The authoritative handoff is an atomically written `receipt.json` with this
shape:

```yaml
schema_version: 1
adapter: local-fixture
task_id: task-123
attempt_id: attempt-20260715T120000Z-abc123
idempotency_key: sha256:...
status: succeeded
requested_stages: [research, draft, qa, final]
artifacts:
  - kind: research_packet
    path: outputs/research/packet.yaml
    sha256: sha256:...
    media_type: application/yaml
needs_review: false
missing_info: []
failure: null
external_run_id: null
started_at: 2026-07-15T12:00:00Z
finished_at: 2026-07-15T12:03:00Z
```

Required receipt invariants:

- `schema_version`, `adapter`, `task_id`, `attempt_id`, `idempotency_key`,
  `status`, `requested_stages`, `artifacts`, `needs_review`,
  `missing_info`, `failure`, `started_at`, and `finished_at` are present.
- `status` is one of `succeeded`, `incomplete`, `rejected`, `failed`, or
  `cancelled`.
- Every declared artifact has a safe relative path, a content hash, and a
  media type. The executor verifies the hash before promotion.
- `failure` is null for successful or incomplete work. For rejected, failed,
  or cancelled work it identifies a stable category and a human-readable
  message. The category includes whether retry is safe.
- `needs_review` is true whenever the output is incomplete, uncertain, or
  requires a human decision. The adapter cannot turn uncertainty into a
  success claim.
- Timestamps are UTC ISO 8601 values. Receipt writes are atomic and a prior
  attempt receipt is never overwritten.

The failure categories are:

| Category | Meaning | Automatic retry |
| --- | --- | --- |
| `contract_invalid` | The planned input cannot be executed as supplied. | No |
| `unsupported` | The adapter cannot perform a requested capability or stage. | No |
| `auth` | Required credentials are missing or rejected. | No, until configuration changes |
| `configuration` | Runtime or environment configuration is invalid. | No, until configuration changes |
| `rate_limited` | The external runtime asked the caller to back off. | Yes, with policy |
| `network` | A transient transport or service reachability failure. | Yes, with policy |
| `provider` | The external runtime returned a provider-side failure. | Only when explicitly retryable |
| `artifact_invalid` | Output is missing, malformed, unsafe, or fails a core validator. | No |
| `cancelled` | An operator or execution policy stopped the attempt. | No by default |
| `internal` | An adapter defect or unexpected runtime failure. | No by default |

### Outcome semantics

- `succeeded`: all requested required outputs were staged and passed the
  applicable core validators. Core may advance lifecycle state and promote
  artifacts.
- `incomplete`: the runtime completed as far as it safely could, but required
  source material, user input, or evidence is missing. Core records the
  review requirement and does not present the work as final.
- `rejected`: the request is disallowed or unsupported. Core records the
  reason and does not retry automatically.
- `failed`: execution did not produce a valid result. Core records the failure
  category and retryability, without changing lifecycle state as though the
  stage succeeded.
- `cancelled`: execution stopped by explicit operator or policy action. Core
  records cancellation and preserves prior artifacts.

## Trust boundary and responsibilities

### Core responsibilities

The core must:

1. Validate the planned task contract before dispatch.
2. Create and own `attempt_id`, `idempotency_key`, and the attempt root.
3. Serialize one execution per task run, or return an explicit conflict.
4. Enforce the staging-root and artifact-path rules.
5. Validate receipt schema, artifact hashes, packet contracts, and stage
   requirements.
6. Own canonical `status.json`, `summary.yaml`, `batch-log.jsonl`, review
   markers, and promotion into knowledge or final output locations.
7. Emit lifecycle events only after receipt and artifact validation.
8. Keep credentials and network policy outside canonical artifacts and logs.
9. Preserve prior attempts so retries are auditable and comparable.

### Adapter responsibilities

An adapter must:

1. Consume only the request and read-only planned inputs it was given.
2. Declare its capabilities, network requirement, and credential requirement
   before execution.
3. Resolve credentials from its runtime environment or approved secret store;
   never copy secret values into the run directory.
4. Perform the external work requested by the selected stages.
5. Write only to its attempt root and produce one atomic receipt.
6. Include all produced artifacts, hashes, missing information, and failure
   details in the receipt.
7. Treat `dry_run` as no external side effects and no durable output claim.
8. Honor the idempotency key when the external runtime supports idempotency,
   and report when it cannot guarantee that behavior.
9. Never promote, overwrite, or reinterpret canonical RDW lifecycle state.

The adapter may use a provider SDK internally in a future implementation, but
that dependency stays behind the adapter boundary and is never imported by
the deterministic core.

## Idempotency, retries, and concurrency

The core derives the idempotency key from the canonical task contract, the
selected adapter, the requested stages, and the relevant input fingerprint.
`attempt_id` is distinct from that stable key: every retry gets a new attempt
directory, while the same logical request retains its idempotency key.

The executor must not silently rerun a task while another attempt is active.
It returns an explicit conflict or waits under a documented policy. A retry
must never overwrite an earlier receipt, and a failed or incomplete attempt
must remain inspectable. Whether an external provider can deduplicate a
request is an adapter capability, not an assumption the core may hide.

Batch concurrency, backoff, and retry budgets are intentionally deferred to
the batch execution ticket. This ADR defines the single-task facts that a
future batch runner must compose.

## Network, credentials, and unsafe effects

The core remains offline and provider-neutral. Network access and external
side effects are explicit properties of an adapter and an execution request,
not implicit behavior of `rdw` planning or validation commands. An adapter
manifest must state whether it needs network access and credentials. The
caller or future executor must opt into those capabilities.

No secret values may appear in task contracts, prompt bundles, receipts,
`batch-log.jsonl`, exception text, or diagnostic output. Provider run IDs and
safe metadata may be recorded when needed for auditability.

Incomplete, uncertain, or rejected work remains visible and reviewable. The
adapter cannot silently substitute fabricated research, suppress missing-info
signals, or promote an external result directly to final copy.

## Local and fake implementation shape

The first implementation should be a deterministic fixture adapter, using the
existing `LocalAdapter` as the compatibility starting point. It will:

1. Read a validated task contract and prompt bundle.
2. Create a deterministic attempt directory from a test-provided fixture.
3. Write a receipt for `succeeded`, `incomplete`, `rejected`, or `failed`
   outcomes without network access.
4. Let the core executor validate the receipt and prove that canonical
   lifecycle files do not change before validation.

This fixture provides contract tests for path safety, hashes, idempotency,
retry preservation, dry-run behavior, and lifecycle handoff before any real
provider runtime is introduced.

## Alternatives considered

### Embed provider SDKs in the RDW core

Rejected. It couples deterministic planning and validation to vendors, makes
network and credentials harder to see, and encourages hidden side effects in
commands that are currently safe to run offline.

### Let adapters write canonical artifacts directly

Rejected. Partial writes, malformed packets, and concurrent attempts could
advance lifecycle state or overwrite trusted outputs before validation.

### Keep the prompt-bundle-only handoff

Useful as the current baseline, but insufficient as an executable contract.
It does not define receipts, attempt isolation, idempotency, failure
semantics, or a validation gate for future runtimes.

### Add hosted orchestration and persistence now

Deferred. It would violate the current local-first product identity and is
not required to prove a safe one-task adapter handoff.

## Migration and verification

The implementation ticket following this ADR will retain the public
`TaskAdapter.run(run_dir)` compatibility seam while introducing typed request
and receipt models plus a receipt validator. Existing
`adapter-<name>.json` summaries may remain as compatibility projections while
`adapter-runs/<adapter>/<attempt_id>/receipt.json` becomes authoritative.

The implementation is complete only when tests prove:

- receipt schema and every outcome category;
- safe relative paths, hash verification, and symlink/traversal rejection;
- stable idempotency keys and new attempt directories for retries;
- no canonical lifecycle mutation before receipt validation;
- incomplete and rejected work remains reviewable and cannot be promoted;
- dry runs make no external side effects or output claims;
- the local fixture works through `rdw adapter run` and in an isolated wheel;
- existing lifecycle, batch, package, and release checks remain green.

## Open decisions deliberately deferred

- Whether a real adapter launches a subprocess, speaks a local protocol, or
  uses a provider SDK internally.
- The exact stage-to-artifact mapping and promotion rules for each stage.
- Packet lineage and merge/conflict behavior for concurrent research updates.
- Batch scheduling, concurrency, retry budgets, and cancellation policy.
- The precise CLI syntax for network and credential opt-in.

These are separate implementation or product tickets. They must not weaken
the boundary defined here.
